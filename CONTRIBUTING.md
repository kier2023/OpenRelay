# Contributing

Thank you for your interest in contributing to this project.

This repository is maintained by a single developer. Contributions are welcome, but they must follow the guidelines below to keep the project stable, maintainable, and easy to review.

---

## Before Contributing

Before opening an issue or pull request, please check the following:

- Search existing issues to see if the problem has already been reported
- Check whether a similar feature request already exists
- Make sure the issue is related to this project specifically

Clear, well-described reports are far more helpful than short or vague ones.

---

## Reporting Bugs

When reporting a bug, please include as much useful information as possible:

- A clear description of the problem
- Steps to reproduce the issue
- Expected behavior vs. actual behavior
- Relevant logs or error messages
- Your environment (Node version, OS, etc.)

If the issue cannot be reproduced, it may not be possible to fix it.

---

## Suggesting Features

Feature suggestions are welcome. However, not every suggestion will be implemented.

When proposing a feature, please explain:

- The problem the feature solves
- Why it would be useful to other users
- A brief idea of how it might work

Features that significantly increase complexity or maintenance cost may be declined.

---

## Pull Requests

Pull requests should be focused, minimal, and clearly described.

Before submitting a pull request:

- Ensure your code runs and builds correctly
- Follow the existing project structure and style
- Keep changes as small and focused as possible
- Include a clear explanation of what the change does

Large, unrelated changes bundled into one pull request are unlikely to be accepted.

---

## Code Style

This project favors:

- Clear, readable code
- Consistent formatting
- Descriptive naming
- Small, focused functions

If you're unsure about style choices, follow the existing patterns used in the codebase.

---

## Project Scope

This project is intentionally kept small and focused.

Changes that introduce unnecessary complexity, large dependencies, or major architectural changes may be declined even if technically sound.

---

## Maintainer Review

All contributions are reviewed by the maintainer.

Pull requests may be:

- Accepted
- Request changes
- Closed if they do not fit the project goals

Please do not take declined contributions personally. Decisions are made based on maintainability and long-term project direction.

## A Final Note

Open source works best when people collaborate respectfully and in good faith. Constructive discussion and thoughtful contributions are always appreciated.
